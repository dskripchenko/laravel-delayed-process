# Проблему «бесконечных спиннеров» в Laravel можно решить проще

Знакомая ситуация: пользователь нажимает «Сгенерировать отчёт», видит спиннер, ждёт 30 секунд… минуту… таймаут. Страница падает, данные теряются, пользователь злится.

Для большинства Laravel-проектов это решается через очереди. Но между «отправить задачу в очередь» и «показать результат пользователю» — огромная пропасть, которую каждая команда заполняет по-своему: WebSocket-серверы, Pusher, кастомные таблицы статусов, ручной polling. Каждый раз — с нуля.

Я столкнулся с этой проблемой на нескольких проектах подряд и в итоге оформил решение в open-source пакет. Возможно, кому-то он тоже окажется полезен.

---

## Laravel Delayed Process

**GitHub:** [dskripchenko/laravel-delayed-process](https://github.com/dskripchenko/laravel-delayed-process)
**Packagist:** [dskripchenko/laravel-delayed-process](https://packagist.org/packages/dskripchenko/laravel-delayed-process)

### Идея

Вместо того чтобы заставлять клиента ждать ответа от тяжёлой операции, сервер мгновенно возвращает UUID. Фронтенд автоматически поллит статус и подставляет результат, когда он готов. Для разработчика — одна строка на бэкенде и одна на фронте.

### Как это выглядит

**Backend — контроллер:**

```php
public function generate(Request $request, ProcessFactoryInterface $factory): JsonResponse
{
    $process = $factory->make(
        ReportService::class,
        'generate',
        $request->integer('user_id'),
        $request->string('format'),
    );

    return response()->json([
        'success' => true,
        'payload' => ['delayed' => ['uuid' => $process->uuid]],
    ]);
}
```

**Frontend — настройка Axios (один раз):**

```typescript
import axios from 'axios';
import { applyAxiosInterceptor } from './delayed-process';

const api = axios.create({ baseURL: '/api' });
applyAxiosInterceptor(api, { statusUrl: '/api/common/delayed-process/status' });
```

Всё. Каждый POST-запрос, который возвращает `delayed.uuid`, теперь автоматически поллится до результата. Для компонентов ничего не меняется — `await api.post(...)` просто возвращает готовые данные, как будто операция выполнилась синхронно.

### Что под капотом

Пакет — не просто обёртка над очередями. Это полноценная система управления долгими процессами:

**Жизненный цикл с 6 статусами.** `new → wait → done | error | expired | cancelled`. Каждый переход — атомарный UPDATE, защищённый от race condition. Процесс не может быть взят двумя воркерами одновременно.

**Автоматический retry.** Если обработчик падает — процесс возвращается в очередь. Количество попыток настраивается. После исчерпания — статус `error` с сохранением сообщения и стектрейса.

**Прогресс 0–100%.** Обработчик может сообщать о прогрессе через `ProcessProgressInterface`. Фронтенд видит актуальный процент на каждом poll-запросе — прогресс-бары из коробки.

**Webhook-уведомления.** При достижении терминального статуса отправляется HTTP POST на `callback_url`. Удобно для интеграций, Telegram-ботов, email-уведомлений.

**TTL и автоэкспирация.** Каждый процесс может иметь `expires_at`. Artisan-команда `delayed:expire` помечает просроченные. Больше никаких «зависших» задач.

**Белый список безопасности.** Только классы из конфига могут быть вызваны как обработчики. Это не фреймворк для запуска произвольного кода — это безопасный конвейер для заранее известных операций.

### Фронтенд — не только Axios

В пакет входят перехватчики для трёх транспортов:

| Транспорт | Подключение |
|-----------|-------------|
| **Axios** | `applyAxiosInterceptor(instance, config)` |
| **Fetch** | `patchFetch(config)` — патчит `window.fetch` |
| **XHR** | `patchXHR(config)` — патчит `XMLHttpRequest` |

Плюс `BatchPoller` для массовых операций — поллит несколько UUID одним запросом.

Защита от зацикливания: заголовок `X-Delayed-Process-Poll` предотвращает перехват poll-запросов.

---

### Per-entity конфигурация

Разные операции — разные требования. Лёгкий сервис идёт в дефолтную очередь, а тяжёлый экспорт — в отдельную с увеличенным таймаутом:

```php
'allowed_entities' => [
    LightService::class,                         // дефолтная очередь
    HeavyService::class => [                     // кастомная
        'queue' => 'heavy',
        'connection' => 'redis',
        'timeout' => 600,
    ],
],
```

### Artisan-команды для продакшна

| Команда | Назначение |
|---------|------------|
| `delayed:process` | Синхронный воркер (для dev или single-server) |
| `delayed:clear` | Очистка терминальных процессов старше N дней |
| `delayed:unstuck` | Сброс «зависших» процессов в статусе `wait` |
| `delayed:expire` | Экспирация процессов с истекшим TTL |

Все команды поддерживают `--dry-run` для безопасного предпросмотра.

---

### События для наблюдаемости

Четыре события жизненного цикла: `ProcessCreated`, `ProcessStarted`, `ProcessCompleted`, `ProcessFailed`. Достаточно подключить Listener — и можно строить мониторинг, метрики, алерты.

---

### Где это может пригодиться

- **Генерация отчётов и экспорт** — PDF, Excel, CSV на десятки тысяч строк
- **Обработка файлов** — конвертация изображений, импорт данных
- **Внешние API** — запросы к медленным сервисам, синхронизация с 1С
- **Массовые операции** — рассылки, пересчёт цен, обновление каталога
- **Любая операция > 5 секунд**, которую пользователь запускает через интерфейс

### Что берёт на себя пакет

- WebSocket-сервер не нужен
- Таблицу статусов проектировать не нужно
- Polling на фронте писать не нужно
- Race conditions обрабатываются внутри
- Retry-логика встроена
